Capabilities

Built for Security

Every design decision prioritizes cryptographic soundness and zero-trust principles.

X3DH Key Agreement

Extended Triple Diffie-Hellman with Ed25519 identity keys, signed prekeys, and one-time prekeys. Asynchronous session setup — Bob can be offline.

Double Ratchet

Forward secrecy and post-compromise security via continuous key rotation. DH ratchet + symmetric KDF chains. Past keys are irrecoverable.

Sealed Sender

Sender identity encrypted via ephemeral DH keypair. The server only knows the recipient — not who sent the message.

Sender Keys Groups

Efficient group E2EE using the Sender Keys protocol. Chain ratchet encryption with Ed25519 signatures on every group message.

Safety Numbers

Signal-style 60-digit safety numbers via 5200 iterations of SHA-512. QR code payloads for in-person key verification.

Zero-Knowledge Server

The relay never sees plaintext, keys, or sender identity. It routes encrypted envelopes and forgets everything. 50K replay dedup cache.

Encrypted Backups

Argon2id key derivation (256MB memory) + XChaCha20-Poly1305 encryption. Passphrase-protected backup and restore.

Message Padding + Replay Guard

PKCS7-style 256-byte block padding hides message length. Multi-layer replay protection with dedup, counters, and server cache.

v2 Hardening

Next-Generation Security

Defense-in-depth with post-quantum cryptography, metadata resistance, and formal protocol verification.

Post-Quantum Hybrid KEM

X25519 + ML-KEM-768 (Kyber) hybrid key encapsulation. If either component is secure, the combined secret is secure. Defends against harvest-now-decrypt-later.

Cryptographic Agility

Immutable cipher suite registry with negotiation. Classical (128-bit) and hybrid PQ (192-bit) suites. Policy enforcement for minimum security levels.

TreeKEM Group Protocol

MLS-inspired tree-based group key agreement with O(log n) update complexity. Per-epoch secrets, Ed25519 signed updates, forward-secret group messaging.

Metadata Resistance

Uniform 4096-byte envelopes make all messages identical in size. Cover traffic, message batching, and timing jitter defeat traffic analysis.

Key Transparency

Merkle tree-based verifiable key directory. Signed tree heads, inclusion proofs, and client-side audit. Prevents silent key substitution by the server.

Shamir Key Splitting

2-of-3 Shamir's Secret Sharing over GF(256). Backup key split across device, server, and printed recovery code. Any 2 shares reconstruct the key.

SecureBuffer

Misuse-resistant key wrapper. Use-after-wipe detection, RAII auto-cleanup scopes, JSON/toString leak prevention. Zero raw key exposure.

Protocol State Machine

Formal session lifecycle: UNINITIALIZED, PREKEY_PUBLISHED, KEY_AGREEMENT, RATCHETING, CLOSED. Invariant enforcement prevents undefined transitions.

Security Posture

We document exactly what is protected and what isn't. No false claims of being "unhackable".

Protected Against

  • Server reading messages (XChaCha20-Poly1305 E2EE)
  • Past message exposure (forward secrecy via Double Ratchet)
  • Ongoing key compromise (post-compromise security)
  • Sender identity leakage (sealed sender)
  • MITM attacks (safety number verification)
  • Replay attacks (client dedup + server 50K cache)
  • Message length analysis (uniform 4096-byte envelopes)
  • Quantum computing threat (post-quantum hybrid KEM)
  • Traffic analysis (cover traffic + batching + jitter)
  • Key directory tampering (Merkle tree transparency)
  • Backup single-point-of-failure (Shamir 2-of-3 splitting)
  • Key material misuse (SecureBuffer with wipe detection)

Remaining Considerations

  • No multi-device support (single keypair per device)
  • PQ KEM uses Kyber wire format placeholder (ready for native drop-in)
  • No TLS in dev mode (production must use wss://)
  • No disappearing messages or remote wipe
  • Needs independent security audit for production

Quick Start

Get Running in 60 Seconds

Clone, install, start. That's it.

# Clone the repository $ git clone https://github.com/BEKO2210/cipherlink.git $ cd cipherlink # Install dependencies $ pnpm install # Start the zero-knowledge relay server $ pnpm dev:server # In another terminal — start the Expo mobile app $ pnpm dev:mobile # Run tests (120 crypto unit tests — 57 v1 + 63 v2) $ pnpm test
